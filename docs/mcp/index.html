MCP quickstart

Protect one MCP tool. See the result immediately.

This is the shortest real path into Attest: wrap an existing MCP server with withAttest(), issue only the scope that tool needs, watch one valid call go through, then block one that should never have been allowed.

What success looks like
You leave this page with one real protected tool call.

By the end of this page, your MCP server is credential-gated, the required scope is discoverable from the server itself, and the same task can flow into audit, evidence export, and verification.

Allowed tool call succeeds A credential with email:send reaches the handler and does the work you expected.
Out-of-scope tool call is blocked A credential missing the required scope gets stopped before the handler runs.
There is proof afterward The same task can be opened in the dashboard, exported as evidence, or verified from a packet.
Activation path
Follow the path that gets you to a real protected MCP tool as fast as possible.
01
Wrap the server you already have Keep your existing tool registration flow. Attest sits in front of it. 
npm install @attest-dev/sdk @modelcontextprotocol/sdk
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { withAttest } from "@attest-dev/sdk/mcp";

const server = new McpServer({ name: "my-tools", version: "1.0.0" });
const protectedServer = withAttest(server, {
  issuerUri: "https://api.attestdev.com",
});

protectedServer.tool("send_email", schema, handler);
02
Let the server declare its own scopes The MCP server should be the source of truth for what each tool actually needs. 
import { getAttestScopes } from "@attest-dev/sdk/mcp";

app.get("/.well-known/attest-scopes", (_req, res) => {
  res.json({ tools: getAttestScopes(protectedServer) });
});
# Create an org and keep the API key shown once
curl -X POST https://api.attestdev.com/v1/orgs \
  -H "Content-Type: application/json" \
  -d '{"name":"acme-corp"}'

# Issue only the discovered scope for send_email
curl -X POST https://api.attestdev.com/v1/credentials \
  -H "Authorization: Bearer att_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "orchestrator-v1",
    "user_id": "usr_alice",
    "scope": ["email:send"],
    "instruction": "Send the weekly digest"
  }'
03
Run one good call and one bad one Make the allowed path succeed, then prove the blocked path fails for the right reason.
Allowed call A tool call to send_email with email:send reaches your MCP handler.
Blocked call A call requiring something like crm:write returns a structured scope violation before the handler runs.
Inspect the evidence trail Open the task in the dashboard, export the evidence packet, or verify it on the public verifier page. 
# After the protected call path is working

+
Manage credentials from inside MCP A separate MCP server that lets agents issue, delegate, revoke, and audit. No SDK code needed.

The middleware above enforces credentials on tool calls. The MCP server below lets an orchestrator agent manage the credential lifecycle itself: issue tokens, delegate to sub-agents, revoke mid-task, and pull audit trails.

{
  "mcpServers": {
    "attest": {
      "command": "npx",
      "args": ["-y", "@attest-dev/mcp-server"],
      "env": {
        "ATTEST_API_KEY": "att_live_...",
        "ATTEST_BASE_URL": "https://api.attestdev.com"
      }
    }
  }
}

14 tools available:

issue_credential list_tasks delegate_credential get_audit_trail revoke_credential get_evidence verify_credential report_action check_revocation report_status request_approval grant_approval get_approval deny_approval

The middleware enforces. The MCP server manages. Use one or both. They work independently.
