Authorization for action-taking agents

Your agents can take action. Can you prove they were allowed to?

When agents send email, update CRM, or touch internal APIs on behalf of people, “we think it was allowed” is not enough. Attest gives every action a signed chain of authority tied back to the human who approved it.

Built for platform, security, and workflow teams shipping agents into real systems, not just chat windows.

// Issue a root credential for your orchestrator
const { token } = await client.issue({
  agentId:     "orchestrator-v1",
  userId:      "usr_alice",
  scope:       ["research:read", "gmail:send"],
  instruction: "summarise inbox and send digest",
});

// Delegate a narrower credential to a sub-agent
const { token: childToken } = await client.delegate({
  parentToken: token,
  childAgent:  "email-agent-v2",
  childScope:  ["gmail:send"],  // must be a subset of parent
});

// Revoke the entire chain instantly
await client.revoke(token);

Every handoff stays tied to a human

The original instruction is bound into the credential chain. Months later, you can still trace an action back to the human request that started it.

Protect the tool boundary

Wrap your MCP server with withAttest() and every tool call has to present a valid credential before your handler runs.

Every child gets less, not more

When an orchestrator hands work to a sub-agent, the child only gets the slice of access it actually needs. You can't accidentally grant more than you started with.

Shut down the whole chain

Revoke the root and every downstream credential dies with it. No child agents quietly keep running after the parent is gone.

Proof, not just logs

Every issue, delegation, denial, and revoke becomes exportable evidence. When security or compliance asks what happened, you have a packet you can verify, not a grep of stdout.

Verification that stands on its own

Credentials and evidence can be verified without calling your backend. The proof still holds when the incident review happens later.

How it works

How Attest fits into a real workflow

Issue a root credential, narrow it at every handoff, enforce it at the tool boundary, and keep proof when something goes wrong.

1

A human approves the work

Your backend calls POST /v1/credentials with the agent ID, user ID, permitted scopes, and original instruction. Attest returns a signed root credential tied back to that approval.

2

The orchestrator hands work downstream

The orchestrator calls POST /v1/credentials/delegate, narrowing scope for each child agent. Every child token carries the chain of authority with it.

3

Every tool call has to prove itself

withAttest() intercepts each tool invocation, verifies the JWT against your org's JWKS, and checks the required scope before your handler runs.

4

Kill the chain when you need to

Call DELETE /v1/credentials/:jti to revoke a token. All descendants are invalidated immediately through cascade revocation.
