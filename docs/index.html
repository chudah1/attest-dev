Your agents take action. Can you prove they were allowed to?

When agents send email, update CRM, or call internal APIs on behalf of people, you need more than "we think it was allowed." Attest gives every action a signed chain of authority tied back to the human who approved it.

Read the docs Try the demo Checking API... 
// Issue a credential tied to a human instruction
const { token } = await client.issue({
  agentId:     "orchestrator-v1",
  userId:      "usr_alice",
  scope:       ["research:read", "gmail:send"],
  instruction: "summarise inbox and send digest",
});

// Delegate a narrower credential to a sub-agent
const { token: child } = await client.delegate({
  parentToken: token,
  childAgent:  "email-agent-v2",
  childScope:  ["gmail:send"],  // subset of parent
});

// Revoke the root and kill the entire chain
await client.revoke(token);

What Attest does

Ties every action to a human

The original instruction is hashed into the credential chain. Months later, you can trace any action back to the human request that started it.

Enforces scope at the tool boundary

Wrap your MCP server with withAttest() and every tool call has to present a valid credential before your handler runs.

Children get less, never more

When an orchestrator delegates to a sub-agent, the child credential is a strict subset. You cannot accidentally grant more than you started with.

Cascade revocation

Revoke the root and every downstream credential dies with it. No child agents quietly keep running after the parent is gone.

Exportable evidence

Every issue, delegation, denial, and revoke becomes a hash-chained audit entry. Export the full evidence packet and hand it to compliance.

Offline verification

Credentials and evidence can be verified without calling your backend. The proof holds even when the incident review happens months later.

How it works

How Attest fits into a real workflow

Issue a root credential, narrow it at every handoff, enforce it at the tool boundary, and keep proof when something goes wrong.

1

A human approves the work

Your backend calls POST /v1/credentials with the agent ID, user ID, permitted scopes, and original instruction. Attest returns a signed root credential tied back to that approval.

2

The orchestrator hands work downstream

The orchestrator calls POST /v1/credentials/delegate, narrowing scope for each child agent. Every child token carries the chain of authority with it.

3

Every tool call has to prove itself

withAttest() intercepts each tool invocation, verifies the JWT against your org's JWKS, and checks the required scope before your handler runs.

4

Kill the chain when you need to

Call DELETE /v1/credentials/:jti to revoke a token. All descendants are invalidated immediately through cascade revocation.
